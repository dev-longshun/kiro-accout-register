# 标准交互工作流

## 第一阶段：需求分析与方案设计

当用户提出**任何修改需求**时（无论大小），你必须按照以下步骤回复：

1. **理解与确认**：简要复述用户的需求，确保理解无误。如有模糊之处，先提问。

2. **方案提出**：根据需求复杂度，提供技术实施方案。
   - **简单修改**（如改颜色、改文案）：说明要改哪个文件的哪个位置，改成什么
   - **复杂功能**：提供至少两个方案（如：方案 A - 快速实现版；方案 B - 性能优化版），并分析优缺点

3. **等待授权**：明确告知用户"确认后我开始执行"或"回复'执行'我就开始改"

## 第二阶段：多轮讨论与优化

- 在此阶段，用户可能会否定方案、提出修改意见或询问细节。
- 你必须保持耐心，根据反馈调整设计方案。
- **注意**：在此阶段，你可以提供伪代码或简短的代码片段来解释逻辑，但**不能**直接修改源文件代码。

## 第三阶段：授权与执行

仅当用户明确回复确认（例如："就用方案 A"、"开始写代码吧"、"执行"）后，你方可进入编码模式：

1. 严格按照商定好的方案编写代码。
2. 应用最佳实践（DRY 原则、注释清晰、错误处理）。
3. 完成后，简要说明修改了哪些文件。

## 文档更新规范

当用户要求更新文档时：

1. **先草拟内容**：在回复中展示要添加/修改的内容（使用代码块）
2. **等待确认**：用户确认后才能写入文件
3. **禁止直接写入**：未经确认不得使用文件写入工具

## Fork 仓库初始化规范

### 触发条件

用户克隆了从 GitHub Fork 的仓库，或提到"fork 仓库"、"初始化 fork"等。

### 核心原则

- `main`（或 `master`）分支保持干净，仅用于同步上游（upstream）
- 所有开发工作在 `main-fork` 分支上进行
- 禁止直接在 `main` 分支上开发

### 初始化流程

1. 确认当前在默认分支上（`main` 或 `master`）
2. 创建并切换到 `main-fork` 分支：
   ```bash
   git checkout -b main-fork
   ```
3. 如果上游仓库尚未添加为 remote，添加 upstream：
   ```bash
   git remote add upstream <上游仓库地址>
   ```

### 同步上游流程

当需要同步上游仓库的最新代码时：

1. 切换到 `main` 分支并拉取上游更新：
   ```bash
   git checkout main
   git fetch upstream
   git merge upstream/main
   ```
2. 切回 `main-fork` 分支，合并 `main`（按分支合并规范执行）：
   ```bash
   git checkout main-fork
   git merge main --no-commit --no-ff
   ```
3. 有冲突则按分支合并规范处理，无冲突则生成 commit 信息

### 禁止事项

- ❌ 直接在 `main` 分支上进行功能开发
- ❌ 跳过 `main-fork` 分支直接从 `main` 创建 feature 分支（feature 分支应从 `main-fork` 创建）
- ❌ 使用 `git push` 将本地 `main` 推送到 origin（`main` 仅用于同步 upstream）
