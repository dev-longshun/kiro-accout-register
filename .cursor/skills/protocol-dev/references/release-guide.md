# 版本发布流程规范

## 触发条件

当用户说"准备发布新版本"、"我要发布新版本"等时，自动执行版本发布流程。

## 🚫 严禁行为

- ❌ 禁止直接修改项目版本配置文件
- ❌ 禁止创建 release commit（用户需要手动提交）

**正确做法**：
- ✅ 只提供版本号的更新建议
- ✅ 生成符合规范的 commit 信息供用户审核
- ✅ 提醒用户手动更新版本号并提交

## 任务一：查找最新版本号

1. 获取当前分支
2. 遍历 git tag 和项目配置文件查找版本号
3. 输出最新版本号

## 任务二：生成更新日志

1. 找到上一个版本发布的 commit
2. 获取从该 commit 到当前 HEAD 的所有提交
3. 按类型分类（feat/fix/perf/refactor）
4. 整合同一功能的多个提交
5. 生成更新日志草稿

### 更新日志格式

```
chore(release): 🔖 发布 v[版本号] - [版本主题描述]

- [功能点1]：[简洁描述]
- [功能点2]：[简洁描述]
```

### 内容整合规则

- 相邻提交如果 scope 相同且描述相关，合并为一条
- 只写最终效果，不写开发过程
- 不包含探索性提交和内部技术细节
- 突出用户价值和体验改进

## 完整工作流程

1. 用户说"准备发布新版本"
2. 查找最新版本号
3. 生成更新日志草稿
4. 展示结果，等待用户确认
5. 用户确认后，提醒手动更新版本号并提交
