# Bug 调试规范 (The "DEBUG-FIRST" Rule)

当用户报告程序中的 Bug 时，**严禁凭直觉猜测问题原因或直接修改业务逻辑**。必须严格遵循以下调试流程。

## 第一步：分析问题现象

1. **明确异常行为**：用户描述的实际发生了什么
2. **明确预期行为**：按照设计应该发生什么
3. **定位问题范围**：确定涉及的代码模块、执行路径和可能的触发条件

## 第二步：提出调试方案（必须等待用户批准）

**🚨 禁止直接向源代码写入调试信息！** 必须先在对话中向用户描述完整的调试方案。

### 调试方案必须包含以下内容

1. **问题分析**：简述对问题的理解和初步判断
2. **调试思路**：解释为什么需要在这些位置添加日志
3. **目标文件清单**：列出需要添加调试信息的文件及具体位置
4. **日志设计**：使用统一前缀便于过滤，明确要记录的变量和状态
5. **明确请求批准**：最后一句必须是"确认后我开始添加调试代码"

## 第三步：插入调试代码（仅在用户批准后执行）

收到用户明确批准后，向代码中添加调试日志。

## 第四步：用户验证与日志分析

1. **请用户复现问题**：运行程序，触发 Bug
2. **收集日志**：用户复制完整的日志输出
3. **分析执行路径**：对比实际与预期，定位异常点
4. **明确问题根因**：基于日志证据得出结论，而非猜测

## 第五步：修复与清理

1. **提出修复方案**（等待用户确认）
2. **执行修复**（用户批准后，保留调试日志）
3. **用户验证**
4. **清理调试代码**（🚨 必须等待用户明确指令，不要主动询问）

## 🚫 严禁行为

- ❌ 未经用户批准直接添加调试日志
- ❌ 不看日志就凭猜测修改逻辑
- ❌ 连续尝试多个"可能的修复"而不通过日志验证
- ❌ 未经用户明确指令直接清理调试代码
- ❌ 在执行修复时顺便删除调试日志
- ❌ 主动询问用户是否清理调试日志

## 核心原则

1. **先分析，后行动**：永远不要跳过分析直接修改代码
2. **日志驱动调试**：所有结论必须基于日志证据
3. **用户批准制**：任何代码修改前都必须获得用户明确批准
4. **清理需指令**：调试日志的清理必须等待用户明确指令
