# 分支合并规范

## 触发条件

用户提出分支合并相关需求，包括但不限于：
- "把 main 合并到当前分支"
- "合并 xxx 分支"
- "同步 main 的最新代码"

## 核心原则

1. **冲突必须停下**：遇到冲突时必须先分析原因、给出方案、等待用户授权后才能解冲突
2. **先谋后动**：合并前必须先分析两个分支的分歧情况

## 工作流程

### 第一步：合并前分析

```bash
git log --oneline <当前分支>...<目标分支> --left-right
git branch -v
```

简要说明两边的分歧 commit 数量和主要改动方向。

### 第二步：执行合并（强制 --no-commit）

```bash
git merge <目标分支> --no-commit --no-ff
```

必须使用 `--no-commit` 和 `--no-ff`。

### 第三步：处理合并结果

#### 无冲突

1. 告知用户合并成功
2. 执行 `git diff --cached --stat` 查看变更
3. 按 commit-guide 的 merge commit 规范生成提交信息

#### 有冲突

**🚨 必须停下来，禁止直接解冲突**

1. 定位冲突文件
2. 分析每处冲突（HEAD vs 目标分支）
3. 给出合并方案
4. 等待用户授权
5. 执行解冲突 + 验证无残留冲突标记
6. 生成 commit 信息

## 禁止事项

- ❌ 不使用 `--no-commit` 直接合并
- ❌ 遇到冲突不分析直接解决
- ❌ 使用 `git merge --abort` 放弃合并（除非用户明确要求）
- ❌ 使用 `git checkout --theirs/--ours` 粗暴解决冲突（除非分析后确认合适）
- ❌ 跳过冲突标记残留检查
